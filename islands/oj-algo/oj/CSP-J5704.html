📜 任务清单 (题目描述)

给定手稿中出现的字符种类数 n，以及每种字符对应的出现频率。

请你利用哈夫曼压缩算法的原理，计算并输出将这些文字信息进行最佳编码后，所需的最小总长度（题目中称为总字节数/总代价）。

羊皮纸压缩规则：

  • 输入的第一行包含一个整数 n，代表手稿里不同字符的种类数量。
  • 接下来的 n 行，每行包含一个字符和一个整数，分别表示该字符的名称以及它在整本手稿中出现的总次数（频率），二者之间用空格分隔。
  • 输出一个整数，代表完全合并压缩后的最小总代价。
📥 魔法输入阵：
(4种字符及其频率)
4
a 45
b 13
c 12
d 16
📤 奇迹输出阵：
(融合后产生的总长度)
152

数据规模限制：

  • 1 < n < 1000
  • 每个字符的出现频率都是正整数，且数值不超过 1,000,000

🧠 魔法道具箱 (核心图解：每次抓取两个最弱的频繁项)

要构建哈夫曼编码，在底层的核心本质上就是施展贪心策略：每次挑选当前频率最低的两个节点，像捏泥娃娃一样把它们融合在一起！

被融合过的新节点，其权值为两个子节点权值的和，并重新投入到待融合的候选池中。在这个重复融合的过程中，被叠加产生的代价值的总和，正是我们梦寐以求的最省空间的哈夫曼树总路径长度。

🗺️ 样例贪心融合步骤图解（输入：a:45, b:13, c:12, d:16） 🗺️
🟢 初始候选池：
c: 12
b: 13
d: 16
a: 45
🔮 第一轮融合： 挑选最小的 12 和 13 合并成新节点 25。
（当前轮次代价消耗：12 + 13 = 25；累计总代价：25）
d: 16
新节点: 25
a: 45
🔮 第二轮融合： 挑选当前池中最小的 16 和 25 合并成新节点 41。
（当前轮次代价消耗：16 + 25 = 41；累计总代价：25 + 41 = 66）
新节点: 41
a: 45
🔮 第三轮融合： 将最后剩下的 41 和 45 合并成根节点 86。
（当前轮次代价消耗：41 + 45 = 86；累计总代价：66 + 86 = 152）
根节点: 86

* 结论显现：每一轮融合时所创造的新节点数值被一路不断地累加，最后得到的答案 25 + 41 + 86 = 152，即为完美哈夫曼树的整体代价值！

✨ 最小堆模型破咒法

如果我们每次都去对数组重新排序，那速度太慢了！为了保证高效率，我们需要借助一个叫做最小堆（Min-Heap / 优先队列）的特殊法宝：

  1. 🔸 注入能量： 把所有字符的频率通通塞进最小堆中。此时堆顶永远是当前数值最小的那一个。
  2. 🔸 双龙戏珠： 只要堆内留存的节点数大于 1，就不断从堆顶弹出两个当前最小的频率值，分别记作 a 和 b。
  3. 🔸 融合归一： 将这两个数字相加，把相加的和 a + b 累加到我们的最终答案计数器中，接着把这个产生的全新和重新塞回最小堆。
  4. 🔸 大功告成： 当堆中最后只剩下一个孤独的根节点时，循环终止，计数器里保存的便是最小总编码长度。
核心秘诀：借助优先队列（最小堆），以 2 对 1 的方式实现无痛实时更新的贪心贪婪模拟

📘 C++ 魔法秘籍 (知识点)

  • 优先队列 std::priority_queue： 在默认情况下，C++ 的优先队列是大根堆（大顶堆）。我们为了让它变成能够随时获取最小值的“最小堆”，必须要加满三项参数配置：priority_queue<int, vector<int>, greater<int>>。
  • 字符摒弃处理： 细心的玩家会发现，在计算总权重代价时，字符本身具体是 'a' 还是 'b' 根本不重要，我们在核心代码里只需要提取输入流中的字符并丢弃它，专注于对整型频率进行统计操作即可。
  • 堆的标准操作： 使用 minHeap.top() 查看堆顶，使用 minHeap.pop() 弹出堆顶，使用 minHeap.push() 放入新值。

💻 C++ 终极法术 (优先队列版)

#include <iostream>
#include <queue>
#include <vector>

using namespace std;

int main() {
    // ⚡ 提速魔法阵，解除输入流束缚
    ios::sync_with_stdio(false);
    cin.tie(NULL);

    int n;
    cin >> n; // 输入不同字符的种类数

    // 🌟 关键道具：建立一个最小堆（小顶堆），用来实时动态维护最小的两个频率
    priority_queue<int, vector<int>, greater<int>> minHeap;

    for (int i = 0; i < n; ++i) {
        char ch;
        int freq;
        cin >> ch >> freq;   // 读取字符（如 'a'）与它对应的出现频率
        minHeap.push(freq);  // 将频率注入最小堆，内部会自动进行排序整理
    }

    int totalBytes = 0;      // 🏆 记录构建哈夫曼树一路上产生的所有代价值总和

    // 🔄 合并循环：只要堆中还有超过1个以上的节点，说明还没有合并到唯一的根节点
    while (minHeap.size() > 1) {
        // 🔮 取出当前全场能量最低、频率最小的两个节点
        int a = minHeap.top(); minHeap.pop();
        int b = minHeap.top(); minHeap.pop();

        // 累计它们融合时产生的代价
        totalBytes += a + b; 

        // 将融合后产生的全新节点重新丢回候选堆池中
        minHeap.push(a + b); 
    }

    // 🏆 输出最终在最节省空间的情况下所压缩出的最小编码总长度
    cout << totalBytes << endl; 

    return 0;
}

🐍 Python 魔法秘籍 (知识点)

  • 内置堆模块 heapq： Python 原生就为我们准备了一个极其高效的堆操作库，名字叫 heapq。与 C++ 不同，Python 的 heapq 默认就是一个纯正的最小堆，完全不需要额外的复杂参数修饰。
  • 堆化法术 heapify： 我们将所有的频率值塞入普通的标准列表后，只需要使用一句 heapq.heapify(freq_list)，Python 就能在惊人的线性时间 O(n) 内直接把一个乱序的列表改造升级为合规的最小堆。
  • 弹栈与压栈： 使用 heapq.heappop(heap) 弹出并返回堆中最小的数据；使用 heapq.heappush(heap, item) 把融合后的新数值重新压入堆里。

💻 Python 终极法术 (heapq内置最小堆版)

import sys
import heapq

def main():
    # 📥 极速快读：一气呵成抓取所有的输入片段
    input_data = sys.stdin.read().split()
    if not input_data:
        return
        
    n = int(input_data[0])
    freq_list = []
    
    # 🔍 从索引位置1开始读取。因为每两个元素组成一组（字符 频率），故跨步为2
    idx = 1
    for _ in range(n):
        # input_data[idx] 是具体字符，由于计算代价与字符内容无关，可以直接忽略
        freq = int(input_data[idx+1])
        freq_list.append(freq)
        idx += 2
        
    # ⚙️ 核心觉醒：在线性 O(n) 时间内将基础列表原地构造成合法的标准最小堆
    heapq.heapify(freq_list)
    
    total_bytes = 0  # 🏆 累加器，用于统计所有的融合权值之和
    
    # 🔄 只要堆内还存留至少两个以上的节点，就不断实施双胞胎合并策略
    while len(freq_list) > 1:
        # 🔮 连拔两筹：弹出当前全场最小的两个频率值
        a = heapq.heappop(freq_list)
        b = heapq.heappop(freq_list)
        
        current_merge_cost = a + b
        total_bytes += current_merge_cost  # 累加这一步合并所付出的代价
        
        # 将合二为一的新能量数值重新放回堆中供后续轮次挑选
        heapq.heappush(freq_list, current_merge_cost)
        
    # 🏆 打印输出羊皮纸手稿压缩后的最优总体积
    print(total_bytes)

if __name__ == '__main__':
    main()

💡 技巧和重点详细总结 (数字化骑士的避坑笔记)

  • 🌟 为什么这道题不需要真正的建树操作？： 很多初学者看到题目叫“哈夫曼编码问题”，就下意识地去写结构体、连左右儿子指针、建一棵真正的二叉树、再写个递归去深度优先搜索跑 0 和 1 的编码。大可不必！ 题目的目标只是求“最优编码后的总代价/总长度”。由于每一次两个子树合并时，两者的频率和恰好会在后续的更高层父节点里被多重复算一次，所以所有非叶子节点（合并产生的中间值）的权值相加，就刚好天衣无缝地等于最终的哈夫曼总编码代价值！ 纯靠模拟合并即可拿到满分。
  • 🌟 哈夫曼思想的经典降维应用： 这种“每次合并两个最小项、并将产生的和计入总代价、重新放回候选池”的通用贪心解题模型，在各大竞赛中经常改头换面出现！比如经典的“合并果子”、“切木板”、“叠纸牌”等等，只要你嗅到了需要最小化合并总损耗的契机，马上把最小堆（优先队列）掏出来，都能达成降维打击。
  • 🌟 绝妙的时间复杂度： 初始建堆或堆化的开销在 O(n) 以内。随后我们总共需要执行 n - 1 轮合并。由于堆的单次弹出和插入操作的时间复杂度都是 O(log n)，因此整体算法的时间复杂度被稳定且近乎完美地锁死在了 O(n log n)。对于题目给出的上限 n < 1000 而言，程序在百微妙之内即可秒杀运行完毕，稳拿满分！
