8.1 分治思想 (Divide and Conquer)

⚔️ 将军的智慧：

试想一下，如果你是一位将军，需要击败一支 10 万人的庞大敌军。硬碰硬显然会全军覆没。聪明的将军会怎么做？他会把敌军分割成 10 个 1 万人的小营地，然后集中兵力，一个一个地解决掉，最后把所有的胜利战果合并起来，赢得整场战争！

这就是计算机科学中最伟大的思想之一：分治法。

一、分治法的基本步骤 (8.1.1)

不管问题看起来多么复杂，分治法总是严格遵循着以下三个极其优雅的动作：

第一步：分解 (Divide) —— 把原本庞大、复杂的大问题，像切蛋糕一样，切成几个长得一模一样，但规模更小的“子问题”。

第二步：解决 (Conquer) —— 继续往下切，直到子问题小到可以直接一眼看出答案（到达递归的边界），然后解决它。

第三步：合并 (Combine) —— 将这些小问题的正确答案，一层一层地拼装回去，最终融合成原先那个大问题的答案。

🛠️ 经典实战：归并排序 (Merge Sort)

“归并排序”是分治法的完美代言人。要把一堆乱序的数字排好，它的策略是：一直对半劈开，直到每份只有一个数字（一个数字天然是有序的）。然后，再两两合并，在合并的过程中把数字按大小排好队伍。

图解 1：归并排序的分治魔法全过程
C++ 归并排序核心逻辑
#include <vector> using namespace std; void mergeSort(vector<int>& arr, int left, int right) { // 1. 递归终止条件 (划分到只有一个元素) if (left >= right) return; // 2. 分解 (Divide)：从中间劈开 int mid = left + (right - left) / 2; // 3. 解决 (Conquer)：交给更小规模的自己去排序 mergeSort(arr, left, mid); mergeSort(arr, mid + 1, right); // 4. 合并 (Combine)：把两个排好序的半区融合成一个 // (此处省略 merge 数组合并的具体实现代码，重点体会分治结构) merge(arr, left, mid, right); }

二、分治与二分的关系 (8.1.2)

很多同学经常把“分治”和“二分”混为一谈，因为它们看起来都在“切东西”。但它们的作战策略其实有本质的区别：

⚔️ 分治 (Divide & Conquer)：我全都要！
虽然把问题切成了两半，但左边和右边都要去解决。最后还要把两边的结果辛辛苦苦地拼凑起来。就像刚刚的归并排序，两边排好序后，最终还要融合。

🎯 二分查找 (Binary Search)：断尾求生！
它也把问题切成两半，但在看了一眼后，直接把错误的那一半扔进垃圾桶！它只会沿着正确的那一半继续往下切。也就是说，二分不需要“合并”，它是在疯狂地缩小包围圈，直到精准锁定唯一目标。

三、时间复杂度分析：递归树与主定理 (8.1.3)

评估分治算法的速度，我们通常画一棵“递归树”。把每一层消耗的时间加起来，就能知道总的时间复杂度。

以归并排序为例，假设给 N 个数字排序：

1. 第一层：把 N 个数分两半，代价是 N（最后合并时需要遍历 N 个数）。

2. 第二层：产生 2 个子问题，每个子问题规模是 N/2。两个子问题合并的代价是 (N/2) + (N/2) = N。

3. 以此类推，无论切分到哪一层，那一层所有小问题的合并代价加起来，始终是 N！

因为每次规模对半砍，总共会砍 log N 层。每一层的代价是 N，所以总时间复杂度就是非常著名的：O(N log N)。

图解 3：归并排序的递归树 (每层工作量都是 N)

📜 扩展：主定理 (Master Theorem) 初步

每次画递归树太麻烦了，大佬们总结出了一个“作弊公式”——主定理。如果一个分治算法的耗时表现为：大问题切成了 A 个小问题，每个小问题的规模缩小为原来的 1/B，而在切分和合并时花费的时间为 O(N 的 D 次方)。

主定理告诉我们：比较 A（碎片的数量）和 B的D次方（每次切分的瘦身比例），谁大谁就决定了最终的时间复杂度！这在后续的算法进阶中，将是你最得力的算力评估工具！
